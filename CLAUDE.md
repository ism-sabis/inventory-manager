# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Build and Run

```bash
go build -o inventory-manager .
./inventory-manager                        # serves on :8000, creates inventory.db, loads data/products.csv
./inventory-manager -port 9000             # custom port
./inventory-manager -db ./data/inv.db      # custom database path
./inventory-manager -data ./mydata         # custom data directory for CSVs
./inventory-manager -remote=false          # localhost only
./inventory-manager -export                # dump SQLite to data/inventory.csv and data/audit-log.csv, then exit
./inventory-manager -import                # load all CSVs from data/ into SQLite (products, inventory, audit-log), then exit
```

No test suite or linter is configured.

## Architecture

Inventory management system with SQLite backend, JSON API, and single-page HTML frontend designed for barcode scanner input.

- `server.go` — main(), flag parsing, HTTP mux setup. Flags: `-path`, `-port`, `-remote`, `-db`, `-data`, `-export`, `-import`.
- `db.go` — SQLite layer using `modernc.org/sqlite` (pure Go, no CGO). Schema migration, all query functions. Items have barcode, SKU, title, image_url, product_url. Barcode-to-SKU resolution via `lookupItem()`.
- `handlers.go` — HTTP handlers for the JSON API. All endpoints accept barcode or SKU interchangeably.
- `csv.go` — CSV import/export. Loads `products.csv` on startup. `-export` dumps inventory.csv, audit-log.csv, orders.csv. `-import` restores from all CSVs.
- `web/index.html` — Single-page frontend with four tabs (Add, Checkout, Orders, Inventory). Vanilla JS + fetch(). Auto-focuses input fields for barcode scanner workflow.

## Data Files (data/)

- `products.csv` — Product catalog (checked into git). Columns: Barcode, SKU, Title, ImageURL, ProductURL. Loaded into SQLite on every server startup.
- `inventory.csv` — Exported snapshot of current stock levels. Generated by `-export`.
- `audit-log.csv` — Exported checkout transaction log. Generated by `-export`.
- `orders.csv` — Exported purchase orders with line items. Generated by `-export`.

The SQLite file is NOT checked in. The CSV files are the portable format: export before shutting down, import to restore on a fresh machine.

## API

All endpoints under `/api/`, JSON request/response. SKUs normalized to uppercase. All endpoints accepting SKU also resolve barcodes.

- `GET /api/items?q=<query>` — search items by SKU, barcode, or title
- `GET /api/items/{sku}` — get item by SKU or barcode
- `POST /api/items` — add stock: `{"sku", "quantity"}`
- `POST /api/checkout` — checkout: `{"sku", "quantity", "project"}`
- `GET /api/checkouts?sku=&project=` — checkout history
- `POST /api/orders` — create purchase order: `{"name", "lines": [{"sku", "quantity"}]}`
- `GET /api/orders?status=open` — list orders, filterable by status
- `GET /api/orders/{id}` — get order with line items
- `POST /api/orders/receive` — receive inventory: `{"line_id", "quantity"}` — adds to stock, auto-closes order when all lines received
- `GET /ping` — health check

## Key Design Decisions

- Barcode and SKU are separate fields. Scanner input resolves barcode → SKU transparently.
- SKUs uppercased on all paths (case-insensitive matching)
- Negative stock allowed (no enforcement on checkout)
- Add stock uses SQLite UPSERT — creates item if new, increments quantity if existing
- Checkout is two-phase in the UI: scan looks up item, then user confirms quantity + project
- Purchase orders track expected inventory: create order with line items, receive against them. Receiving adds to stock. Order auto-closes when all lines fully received. Items show `on_order` quantity from open orders.
